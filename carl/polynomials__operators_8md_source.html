4 The classes used to build polynomials are (almost) fully compatible with respect to the following operators, that means that any two objects of these types can be combined if there is a directed path between them within the class hierarchy.

23 ## Comparison operators
24 All of these operators are defined for all combination of types.
25 We use the following ordering:
26 - For two variables `x` and `y`, `x < y` if the id of `x` is smaller then the id of `y`. The id is generated automatically by the VariablePool.
27 - For two monomials `a` and `b`, we use a lexicographical ordering with total degree, that is `a < b` if
28  - the total degree of `a` is smaller than the total degree of `b`, or
29  - the total degrees are the same and
30  - the exponent of some variable `v` in `a` is greater than in `b` and
31  - the exponents of all variables smaller than `v` are the same in `a` and in `b`.
32  - The intuition is that the monomials are considered as a sorted product of plain variables.
33 - For two terms `a` and `b`, `a < b` if
34  - the monomial of `a` is smaller than the monomial of `b`, or
35  - the monomials of `a` and `b` are the same and the coefficient of `a` is smaller than the coefficient of `b`.
36 - For two polynomials `a` and `b`, we use a lexicographical ordering, that is `a < b` if
37  - `term(a,i) < term(b,i)` and
38  - `term(a,j) = term(b,j)` for all `j=0, ..., i-1`, where `term(a,0)` is the leading term of `a`, that is the largest term with respect to the term ordering.

91 ## Implementation
92 We follow a few rules when implementing these operators:
93 - Of the comparison operators, only `operator==` and `operator<` contain a real implementation. The others are implemented like this:
94  - `operator!=(lhs, rhs)`: `!(lhs == rhs)`
95  - `operator<=(lhs, rhs)`: `!(rhs < lhs)`
96  - `operator>(lhs, rhs)`: `rhs < lhs`
97  - `operator>=(lhs, rhs)`: `rhs <= lhs`
98 - Of all `operator==`, only those where `lhs` is the most general type contain a real implementation. The others are implemented like this:
99  - `operator==(lhs, rhs)`: `rhs == lhs`
100 - They are ordered like in the list above.
101 - Operators are implemented in the file of the most general type involved (either an argument or the return type).
102 - Operators are not implemented as friend methods. Those are usually only found by the compiler due to ADL, but as we need to declare `operator+(Term, Term) -> MultivariatePolynomial` next to the MultivariatePolynomial, this will not work. If a friend declaration is necessary, it will be done as a forward declaration.
103 - Overloaded versions of the same operator are ordered in decreasing lexicographical order, like in this example:
104  - `operator(Term,Term)`
105  - `operator(Term,Monomial)`
106  - `operator(Term,Variable)`
107  - `operator(Term,Coefficient)`
108  - `operator(Monomial,Term)`
109  - `operator(Variable,Term)`
110  - `operator(Coefficient,Term)`
111 - Other versions are below those.

113 ## Testing the operators
114 There are two stages for testing these operators: a syntactical check that these operators exist and have the correct signature and a semantical check that they actually work as expected.
115 
116 ### Syntactical checks
117 The syntactical check for all operators specified here is done in `tests/core/Test_Operators.cpp`.
118 We use `boost::concept_check` to check the existence of the operators. There are the following concepts:
119 
120 - `Comparison`: Checks for all comparison operators. (`==`, `!=`, `<`, `<=`, `>`, `>=`)
121 - `Addition`: Checks for out-of-place addition operators. (`+`, `-`)
122 - `UnaryMinus`: Checks for unary minus operators. (`-`)
123 - `Multiplication`: Checks for out-of-place multiplication operators. (`*`)
124 - `InplaceAddition`: Checks for all in-place addition operators. (`+=`, `-=`)
125 - `InplaceMultiplication`: Checks for all in-place multiplication operators. (`*=`)
126 
127 ### Semantical checks
128 Semantical checking is done within the test for each class.
